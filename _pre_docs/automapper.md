# Automapper 阶段一：自动铺墙

## 核心理念

**草图只是参考，一切以生成的地图实物为准。**

算法优先级：
1. **结构正确**：生成的墙壁必须闭合、横平竖直、严格对齐等轴测网格。不允许断壁、缺口、孤立碎块。
2. **贴近草图**：在结构正确的前提下，尽可能逼近用户手绘意图。草图偏了就"掰直"，算法说了算。

> 草图是"意图指引"，最终形态由算法的网格逻辑接管。

---

## 工具定位

**辅助型关卡编辑器**——开发者画草图，程序自动砌墙，生成的 `.map` 文件直接用官方 MapEdit.exe 打开做后续编辑。

## 等轴测投影公式

对任意逻辑格点 $(gx, gy)$，物理坐标为：

$$posX = (gx - gy) \times 40.0 + shiftX$$
$$posY = (gx + gy) \times 28.0 + shiftY$$

- 步进常量来自逆向实测：`stepX = 40.0`，`stepY = 28.0`
- `shiftX / shiftY` 为偏置量，用于将建筑居中在地图物理边界内

---

## 墙体图块字典

| 类型 | VID | 说明 |
|------|-----|------|
| 走向 A 斜墙 (`/`) | `601` | 左上→右下方向，三者仅贴图变体不同 |
| 走向 B 斜墙 (`\`) | `602` | 右上→左下方向，三者仅贴图变体不同 |
| 转角立柱 | `604` | 所有转角、端头、交叉处使用 |

- 所有墙体和立柱的 `direction` 字段固定为 `32`
- 铺设时在同走向变体中**随机选取**，以避免大面积重复的单调感

---

## 关键偏置规律

- **走向 A 墙 (`601` 系列) 与立柱 (`604`)**：偏置 `(0, 0)`，直接使用投影公式结果
- **走向 B 墙 (`602` 系列)**：偏置 `(-40.0, 0.0)`，即 `posX` 需额外左移 40 像素
- **立柱**：在任何场景（转弯、交叉、断面）下偏移量恒为 `(0, 0)`，不随所连接墙体漂移

## 前端交互：从“位图涂鸦”到“矢量线段”

前端画板不记录用户高频的鼠标移动事件（像素轨迹），而是将玩家的一次拖拽操作**压缩并纠偏**为标准矢量线段，作为后端算法的统一输入。

**核心处理逻辑**：
1. **获取端点**：监听用户的鼠标拖拽，仅提取起始点 A 和松开点 B 的屏幕坐标。
2. **网格吸附**：将 A 和 B 分别换算并四舍五入到最近的逻辑网格坐标 $(gx_1, gy_1)$ 和 $(gx_2, gy_2)$。
3. **强制拉直（数据纠偏）**：
   - 引擎限制墙壁必须横平竖直。计算两点在 X 和 Y 轴上的逻辑跨度：`dx = |gx2 - gx1|`，`dy = |gy2 - gy1|`。
   - 对比跨度，强行剥离较小的偏移量，将斜线“掰直”。
   - 若 `dx > dy`（玩家意图主要为横向），则强制令 `gy2 = gy1`。
   - 若 `dy >= dx`（玩家意图主要为纵向），则强制令 `gx2 = gx1`。
4. **数据输出**：将处理后的纯净数据组装为线段 `((gx1, gy1), (gx2, gy2))`，追加到 `wall_segments` 列表中。

**架构优势**：
- **数据极度压缩**：成百上千个抖动的鼠标采样点被降维成几个数字，大幅节省资源。
- **前后端彻底解耦**：底层铺墙算法无需关心前端交互产生的任何脏数据，只负责处理纯净的线段列表。
- **极简状态管理**：撤销/重做（Undo/Redo）无需处理像素擦除，只需对 `wall_segments` 数组执行 `pop` 或 `push` 即可。

## 铺墙判定逻辑（伪代码）

```
输入: wall_segments — 墙线段列表，每条线段 = [(gx1,gy1), (gx2,gy2)]
      线段只能沿 gx 轴（走向A）或 gy 轴（走向B）延伸

第一步：展开所有线段，记录每个格点的走向
  遍历每条线段：
    记录起点和终点为"端点"
    若 gy 不变 → 走向 A，沿 gx 逐格标记
    若 gx 不变 → 走向 B，沿 gy 逐格标记
    结果：grid_walls[(gx, gy)] = {走向集合}

第二步：逐格判定——放立柱还是放直墙？
  对每个格点 (gx, gy)：
    放立柱的条件（满足任一即可）：
      ① 是线段端点
      ② 同时有 A 和 B 两个走向交汇（转弯/交叉）
      ③ 沿走向方向的相邻格没有同走向的墙（悬空截断）

    否则：放直墙
      走向 A → 用投影公式，偏置 (0, 0)，VID 从 [601, 609, 610] 随机
      走向 B → 用投影公式，偏置 (-40, 0)，VID 从 [602, 603, 612] 随机

第三步：输出精灵列表
  每个精灵 = { vid, posX, posY, posZ: 0, direction: 32 }
```

## 已确定的设计决策

| 决策项 | 结论 |
|--------|------|
| 数据管线 | 跳过 JSON 中间层和 `map_tool.py pack`，**直接写二进制 `.map`** |
| 地图基底 | 始终基于 `empty.map` 模板从零生成，**不支持在已有地图上追加** |
| objectId | 简单自增计数器（从 1 开始），全局唯一即可 |
| 最小房间 | 1×1 格 = 4 面墙 + 4 根立柱 |
| terrain | 阶段一**不动** `height_map`，碰撞问题后续阶段处理 |
| 验证标准 | 在 MapEdit 中肉眼验证，保证房间完整闭合即可 |
| mapSize | **用户指定** `mapSizeX / mapSizeY`，画板尺寸与之一致 |
| 房间形态 | 必须是**封闭实体**，形状不限（L形、T形、拼接矩形均可） |
| SPRD | 墙壁无 behavior，cmdCount=0，itemCount=0，army=0。除 objectId 外全部固定 |
| 草图精度 | 用户草图 ≠ 最终墙壁，算法**尽可能贴近轮廓**但以网格结构正确为优先 |
| 开发策略 | **Demo 先行**——先用硬编码墙线段验证算法+二进制管线，画板 UI 延后 |

## Demo 开发路径

画板 UI 是锦上添花，算法和二进制写入才是地基。最快的验证路径：

```
硬编码墙线段 → 铺墙算法 → 精灵列表 → 二进制 .map → MapEdit 打开验证
```

### Demo 脚本职责

一个 Python 脚本，包含三个独立模块：

1. **墙体生成器** (`wall_builder`)
   - 输入：墙线段列表 `[((gx1,gy1), (gx2,gy2)), ...]`
   - 输出：精灵列表 `[{vid, posX, posY, posZ, direction, army}, ...]`
   - 核心逻辑：展开线段 → 判定立柱/直墙 → 投影公式计算物理坐标

2. **二进制写入器** (`map_writer`)
   - 输入：精灵列表 + mapSizeX/Y
   - 输出：`.map` 二进制文件
   - 核心逻辑：读 `empty.map` 模板 → 拼 SPR/SPRD 段 → 回填 size

3. **测试用例** (`main`)
   - 硬编码一个 5×5 正方形房间的墙线段（与 `room_sample.json` 等效）
   - 调用上述两个模块生成 `test_output.map`
   - 用 `as1_map_unpack.py` 解包验证结构正确性

### 验证方法

## 二进制直写管线

跳过 JSON 中间层，直接将精灵数据拼成二进制 `.map` 文件。`empty.map`（228 字节）提供固定段的模板数据。

### .map 文件段结构回顾

```
FileHeader (12B)  →  "RES " + mapSize(uint32) + "MAP "
GRPH (44B)        →  图形环境参数（阴影、伽马、风向），固定长度
HEAD (48B)        →  地图尺寸、摄像机位置、版本号，固定长度
SPR  (可变)       →  精灵位置数据。每条记录 28B，末尾 0xFFFFFFFF
SPRD (可变)       →  精灵详细数据。墙壁精灵全部为固定模板（见下）
PLAY (40B)        →  玩家 ObjectId 映射，固定长度
GROU (可变)       →  分组信息。阶段一无分组
```

### 拼装流程（伪代码）

```
1. 读取 empty.map 的 GRPH 段（44B）→ 原样复用
2. 读取 empty.map 的 HEAD 段（48B）→ 写入用户指定的 mapSizeX / mapSizeY
3. 构建 SPR 段：
   - 段头: "SPR " + sprSize + options(0x80000000) + packedDiff(0) + subCount(1) + sprInfoSize
   - 逐条写入 SpriteRecord (28B each):
       objectId(uint32) + vid(int32) + posX(float) + posY(float) + posZ(float=0) + direction(uint32=32) + army(int32=0)
   - 末尾: terminator 0xFFFFFFFF
   - sprInfoSize = spriteCount × 28 + 4
4. 构建 SPRD 段：
   - 每个墙壁精灵的 sub 完全固定（12B）:
       subSize(12) + objectId + cmdCount(0) + itemCount(0)
   - 末尾 terminator sub: subSize(4) + 0xFFFFFFFF
   - subresourceCount = spriteCount + 1
5. 原样写入 PLAY 段（40B，全零玩家ID）
6. 构建空 GROU 段: "GROU" + size(20) + options + packedDiff(0) + subCount(1) + infoSize(4) + 0xFFFFFFFF
7. 回填 FileHeader: mapSize = 文件总字节数 - 8
```

### 单条 SpriteRecord 的二进制布局 (28 字节, Little-Endian)

```
偏移  类型     字段          墙壁固定值
0x00  uint32   objectId      自增ID
0x04  int32    vid           601/602/604/...
0x08  float    posX          投影公式计算
0x0C  float    posY          投影公式计算
0x10  float    posZ          0.0
0x14  uint32   direction     32
0x18  int32    army          0
```

### 单条 SPRD Sub 的二进制布局 (墙壁专用, 12 字节)

```
偏移  类型     字段          固定值
0x00  uint32   subSize       12
0x04  uint32   objectId      与 SPR 中对应记录一致
0x08  uint32   cmdCount      0
0x0C  uint32   itemCount     0
```

## 物理边界居中与网格对齐（Grid Snapping）

### 纯数学居中偏置

生成前需计算偏置量，确保所有精灵落在地图合法区间 `[0, mapSizeX]` × `[0, mapSizeY]` 内：

```
1. 先以 shiftX=0, shiftY=0 算出所有精灵的包围盒
2. raw_shiftX = (mapSizeX - 包围盒宽) / 2 - 包围盒最小X
3. raw_shiftY = (mapSizeY - 包围盒高) / 2 - 包围盒最小Y
```

### 核心对齐基准（Grid Snapping Base）

官方编辑器内置了两种对齐模式：

#### 1. 正交网格对齐（快捷键 `s`）
- **功能**：按格点移动光标（`поклеточного смещения курсора`）。专门用于铺设墙壁、立柱、独立物件。
- **底层数学本质**：
  引擎的底层网格尺寸为 `STEP_X = 40`，`STEP_Y = 28`。
  所有坐标被强制限制在：`posX ≡ 20 (mod 40)` 且 `posY ≡ 14 (mod 28)`。
  这绝对不是一个随机的“魔法数字”，而是**网格的中心点位置**。即 `(STEP_X / 2, STEP_Y / 2)`。
  在等轴测地图中，绝对坐标 `(0,0)` 是左上角边界，而引擎规定所有物件的坐标定位锚点必须落在“逻辑格网的几何中心”，因此产生了全局的 `+20, +14` 偏置。

#### 2. 奇数行半格错位对齐（窗格按钮）
- **功能**：将所有奇数行偏移半个网格。专门用于铺设地面（Terrain/Floor）。
- **数学逻辑**：
  在 `s` 网格的基础上，根据 Y 轴行号 `RowY = (posY - 14) / 28` 决定 X 坐标：
  - `RowY` 是**偶数**：`posX ≡ 20 (mod 40)`
  - `RowY` 是**奇数**：`posX ≡ (20 + 40/2) ≡ 0 (mod 40)`
- **当双开时**：形成严丝合缝的等轴测菱形交错网格（Staggered Isometric Grid），地砖行与行之间错开半个身位嵌套。

### 墙壁偏移的几何规律与长墙壁

1. **走向 B（`\`）的 `-40` 偏移推算**：
   我们发现 `\` 走向的墙壁在渲染时必须手动 X 坐标 `-40`。这并非需要硬编码查表（VID），而是由几何规则决定的。
   - `走向 A (/)` 是沿 Y 轴延伸，贴图锚点自然贴合当前格点。
   - `走向 B (\)` 是沿 X 轴延伸，在视觉透视上，它的贴图其实是挂在相邻网格的左边缘。为了对齐当前的立柱中心，它的物理坐标必须强制向左平移一个单位跨度（`-STEP_X`，即 `-40` 像素）。
   未来算法实现中，仅通过判断“边”的走向（平行于 X 还是 Y 轴），即可自动应用该偏移。

2. **双倍长墙壁的吸附表现**：
   游戏中存在双倍长度（占 2 个格距）的长墙壁。
   因为全局引擎的吸附网格 `s` 依然是固定的单格步长（40×28），当你拿着长墙壁在编辑器里拖动时，它依然会吸附在上述的 `(20, 14)` 的基准点上。
   这就导致了：如果按照步长 `1` 连续放置长墙壁，后一面墙会有一半直接“插/叠”进前一面墙里。要完美拼接长墙壁，算法的遍历步长必须调整为 `2`（即 `gx += 2` 或 `gy += 2`）。
