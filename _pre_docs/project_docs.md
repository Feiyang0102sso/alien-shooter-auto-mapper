# Auto Mapper 项目规范与结构规划

## 技术栈规范
- **算法层**：必须优先使用 **C++** 编写（包括核心逻辑、线段处理、坐标换算、网格判定等）。除非遇到极度麻烦且影响效率的特例情况，才考虑降级使用 Python 处理算法逻辑。
- **UI与胶水层**：使用 Python 编写。主要负责实现前端交互（画板等）、调用 C++ 编译出的核心库，以及进行一些非核心的脚本化操作。

## 核心难题与坐标系解耦 (坐标系歧义与物理引擎重构)

### 1. 最大的难题是什么？(前端 UI 的 x, y 代表什么？)
在 UI 前端画布上画线时，前端发送给后端的坐标 `(gx, gy)` **并不是一个统一的抽象画布网格数**，而是**绝对绑定于当前所选墙体特征（WallProfile）自身的逻辑网格数**！
- 如果你画的是**标准墙壁 (40x28)**，线段 `x=0` 到 `x=3` 意味着 $3 \times 40 = 120$ 物理等距像素长度。
- 如果你画的是**实验室墙 (90x64)**，线段 `x=0` 到 `x=3` 意味着 $3 \times 90 = 270$ 物理等距像素长度。

这就意味着，前端虽然在一张 Canvas 上把它们画在了一起，但在后端的逻辑空间里，**不同材质的墙体生存在完全不同的数学坐标系中**。它们的基底向量（步长 `step_x/step_y`）不同，由于居中对齐导致的投影原点 (`shift_x/shift_y`) 也不同。
如果后端试图在由 `(gx, gy)` 组成的“逻辑网格”上运行 Flood Fill (区域泛滥算法) 寻找闭合房间，就会遭遇灾难性的拓扑撕裂：不同的网格根本拼不到一块去，导致铺地板时出现漏铺、多铺和严重错位。

### 2. 解决方案：物理视角的降维打击 (全局物理 Flood Fill)
既然逻辑坐标系是分裂的，我们就**彻底抛弃在逻辑网格上做碰撞检测**，而是降维到“物理像素世界”来做：
1. **多级栅格化 (Bresenham 画线)**：不管是什么墙体，拿到 `(gx, gy)` 后，立刻调用 `to_physical` 将其首尾两端转换为准确的物理像素坐标 `(px, py)`。随后在一张高精度的**物理布尔大网格**（如每 5x5 像素为一个 Cell）上把墙体连成“物理实线”。
2. **全局 Flood Fill**：算法直接在这个物理大网格的边缘开始灌水（Flood Fill），没被灌到的地方就是“墙内 (INSIDE)”，灌到的地方就是“墙外 (OUTSIDE)”。
3. **独立采样对齐**：当地板或天花板要铺设时，它们按照自己独有的步长（如 80x56）遍历自己的逻辑网格，算出中心物理点 `px, py`，然后去向物理网格“打听”自己到底属于 INSIDE 还是 OUTSIDE。

这种做法实现了彻底的**坐标系解耦**，不仅完美兼容任何尺寸奇葩的墙体混搭，更是百分之百还原了 UI 界面上用户肉眼所见的拓扑结构。
