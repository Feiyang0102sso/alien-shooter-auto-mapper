# Auto Mapper 坐标系统原理解析

在 `Auto Mapper` 的 UI 渲染与交互逻辑中，为了实现带有等距视角（Isometric）的地图效果，并同时支持流畅的缩放与拖拽，系统中实际上**同时运行着三套不同的坐标系**。

为了更直观易懂，我们把它们替换为更具体的称呼：

## 1. Grid 坐标 (逻辑坐标 Logical Coordinates: `gx`, `gy`)

- **具体含义**：**即用户在 UI 上直接画画的那部分纯粹网格。**
- **作用**：这是最底层、最纯粹的业务数据。当你在屏幕上点两下画出一条墙壁（比如从第 0 格连到第 5 格），系统记录在 `self.segments` 里的就是这个 Grid 坐标。
- **特征**：它在数据层面上是平的，不带任何缩放、拖拽或者倾斜视角的杂质。最后喂给 C++ 引擎去运算的，也就是这份干干净净的 Grid 坐标。

## 2. Map 坐标 (物理坐标 Physical Coordinates: `px`, `py`)

- **具体含义**：**底层引擎中真实的标准 2D 坐标系（X 向右延伸，Y 向下延伸）。**
- **作用与真相**：注意，Map 坐标系本身其实是**正正方方的**，完全没有变歪！我们眼睛看到的“歪”和“立体感”，纯粹是**美术模型（贴图）带来的视觉魔术**。
- **“手工敲打”出的排版公式**：
  在系统层面，将 Grid 换算到 Map 的过程本质上就是一个“排版”过程——系统需要确切地知道，把一张带着立体视角的墙壁贴图放在哪个像素点上，才能不留缝隙地拼出地图。

  ```python
  px = (gx - gy) * 40.0 + grid_shift_x
  py = (gx + gy) * 28.0 + grid_shift_y
  ```

  这个公式分为两个部分互相配合：
  
  1. **步伐间距的乘法：`(gx - gy) * 40.0` 和 `(gx + gy) * 28.0`**
     - 因为模型图像是有固定比例的，经过手工测试发现：顺着 `X` 走一格，下一张图要往右下角挪 `(40, 28)` 像素；顺着 `Y` 走一格，下一张图要往左下角挪 `(-40, 28)` 像素。
     - 这一步相当于泥瓦匠手里的卷尺，精确算出了两块瓷砖之间的像素距离。
     
  2. **绝对对齐的加法（找中心起点）：`grid_shift_x` 与 `grid_shift_y`**
     ```cpp
     // 引擎内部真实的起点平移代码
     float grid_x = std::round((raw_shift_x - 20.0f) / 40.0f);
     shift.x = grid_x * 40.0f + 20.0f;
     ```
     - 第一步，先把整张网格的大本营（即起始点）搬到画布的正上方中间位置，防止朝某个方向画图时直接飞出屏幕左侧。
     - 第二步，最关键的**半步长吸附（Snap）**：它不使用随机的小数起点，而是强行将起点“咔哒”一声吸附到 `N * 40 + 20` 这种固定的半格卡槽里（20 是 40 的一半）。这样做是为了迎合贴图自带的中心点（Anchor Point），保证最后算出来的每个坐标点，能刚好落在显示器的物理整数像素点上。如果没有这一步，画出来的图像可能会糊掉（出现次像素模糊）或者带毛边。

## 3. 屏幕像素坐标 (Screen Coordinates: `sx`, `sy`)

- **具体含义**：**你眼睛真正在显示器（UI 画布）上看到的像素点位置。**
- **作用**：不管底层的 Map 地图有多庞大，用户的显示器窗口只有那么大。这一层主要负责处理“玩家视角”，它往 Map 坐标上又叠加了三样东西：
  1. **缩放 (Zoom)**：你用鼠标滚轮放大的倍数。
  2. **居中 (Center Offset)**：确保无论地图多大，都能在一开始老老实实呆在窗口正中央的偏移量。
  3. **拖拽 (Pan)**：你按住鼠标中键在地图上四处拖拽产生的距离。
- **转换公式**：
  ```python
  sx = px * 缩放倍率 + 居中偏移 + 拖拽偏移
  sy = py * 缩放倍率 + 居中偏移 + 拖拽偏移
  ```

## 4. 这三套系统是怎么合作的？(Data Flow)

它们三者就像同声传译一样，在每次你点击或者滑动鼠标时疯狂地互相翻译：

### 📥 鼠标输入时 (屏幕像素 ➡️ Grid 坐标)
当你想在地图上画个点时：
1. 你的鼠标点击了屏幕，程序获取了屏幕像素坐标 **(sx, sy)**。
2. 程序先脱去“缩放和拖拽”的外衣，算出来你点在了底层地图的哪里，这就变成了 Map 坐标 **(px, py)**。
3. 接着程序解开“等距立体透视”的魔法，把斜着的地图反算回横平竖直的格子，变成了 Grid 坐标 **(gx, gy)**。
4. 因为你手抖点出的格子坐标带有小数，程序最后做了一步“吸附”，把它四舍五入对齐到最近的整数格子上。

### 📤 画面渲染时 (Grid 坐标 ➡️ 屏幕像素)
当程序要在屏幕上画你刚刚存下的线段时：
1. 取出你之前保存的纯净网格数据，即 Grid 坐标 **(gx, gy)**。
2. 经过数学公式，把四方格子铺平到斜着的 2.5D 地板上，变身为 Map 坐标 **(px, py)**。
3. 乘上你当前的缩放倍率，再加上你刚才拖拽和居中的位置，计算出它最终究竟应该落在窗口屏幕的哪几个像素上 **(sx, sy)**，然后指挥 UI 画笔画出来！

## 5. 贴图物理锚点偏移 (Sprite Anchors & Offsets)

虽然通过上述公式，我们已经精确算出了逻辑网格中每一个交汇点（Corner）的底层 Map 坐标 `pos(gx, gy)`。但在真实的 2.5D 游戏开发中，不同画师、不同尺寸的素材贴图往往存在一个“原点不对齐”的深坑。

为了彻底解决此问题，并实现贴图系统的完全解耦，我们在核心系统中引入了 `WallProfile` 架构，为构成一面墙的三种不同组件配置了独立的物理像素偏移量：

- **`offset_a` (`/` 方向墙，内部代号 `dir_a`)**
- **`offset_b` (`\` 方向墙，内部代号 `dir_b`)**
- **`offset_p` (立柱，内部代号 `pillar`)**

### 为什么需要这些单独的 Offset 属性？
在等距视角（Isometric）中，地图由一个个倾斜的菱形网格拼接而成。我们在逻辑网格中画墙，实际上是在沿着菱形的边缘放置墙壁贴图。
以网格上的同一个坐标点 `(gx, gy)` 为例：
1. **`dir_a` 墙**：代表的是从 `(gx, gy-1)` 延伸到 `(gx, gy)` 的墙壁（/ 走向）。
2. **`dir_b` 墙**：代表的是从 `(gx-1, gy)` 延伸到 `(gx, gy)` 的墙壁（\ 走向）。
3. **`pillar` 立柱**：立柱直接矗立在 `(gx, gy)` 这个角上，用来掩盖两面墙交汇产生的接缝。

在理想的数学模型下，这三个贴图都应该把自己的底部中心位置当作绝对的坐标原点。如果我们把它们都强行塞入算出来的底层 Map 坐标 `pos(gx, gy)` 里，它们就会自动完美咬合拼合。

### 残酷的现实：游戏画师的随意性
但在实际的《孤胆枪手》引擎数据里，事实并非如此：
- **对于标准墙 (Standard Wall, 40x28)**：画师规定 `/` 墙（601）和立柱（604）的锚点在底端。但对 `\` 墙（602），画师却把锚点定在了贴图的左端。所以我们必须在放置 `dir_b` 时，将其向左偏移 `-40` 像素（即配置 `offset_b_x = -40`），才能让它乖乖回到原处和别的墙缝合。
- **对于实验室厚墙 (Lab Wall, 90x64)**：这是极其庞大且带有厚度的模型。通过对官方游戏原生关卡存档的二进制数据逆向还原，我们发现：官方为了让这套厚墙视觉上完美拼接，竟然强行在系统级，将 `/` 墙（651）的绘制坐标向右偏了整整 90 像素（即配置 `offset_a_x = +90`），而对 `\` 墙和立柱则一点没偏。

### 架构意义与价值
这就是 `dir_a / dir_b / pillar` 偏移属性的核心意义：**它们充当了“底层纯粹逻辑坐标”与“粗糙美术贴图”之间的缓冲层。**

不论未来引入外部的贴图素材多么奇葩、锚点多么不规范，我们底层的坐标数学引擎始终输出绝对纯粹、严密推导出来的 `pos(gx, gy)` 绝对坐标。而在最终生成给游戏引擎的 MAP 文件时，我们通过配置 `WallProfile`，独立修正这画师留下的烂摊子。这样既保全了核心算法的优雅，又实现了对游戏复杂素材的高可复用性。

### 附录：如何逆向推算出未知的 Offset？（以 Lab Wall 为例）

当我们拿到一套全新的地形素材（如 650/651/652）但完全不知道它的锚点规律时，最好的办法是**解析官方游戏自带的地图存档（JSON 格式的数据提取）来进行数学推导**。

**推算核心思路与过程：**
1. **寻找坐标重合点（Corner Pivot）作为基准**：
   在官方数据中，寻找多面墙相互交汇的角。因为视觉上完美闭合的墙角，必有一根立柱（652）覆盖在上面。
   *查阅 JSON 数据发现：在一个菱形的右侧闭合外角，立柱的物理绝对坐标被定在了 `(1012, 400)`。*

2. **确定立柱的锚点定位**：
   既然这个网格交汇点（Corner）在底层逻辑坐标 `pos(gx, gy)` 就是 `(1012, 400)`，这就说明**立柱本身是不带任何偏差的锚点基准**。因此得出结论：立柱 `offset_p = (0, 0)`。

3. **逆推相连墙壁的错位差值**：
   紧接着，在这组官方数据中，去寻找同样交汇于此的墙壁坐标。
   - **推导 `\` 方向墙（dir_b, 650）**：在数据中，紧紧依附在这个角上的 `\` 墙，它的坐标竟然**也是 `(1012, 400)`**！这从数学上证明了，该方向的墙只要直勾勾地下在逻辑锚点上，就能完美对接。所以得出：`offset_b = (0, 0)`。
   - **推导 `/` 方向墙（dir_a, 651）**：这是最关键的突破口。在数据中，紧贴该角的 `/` 墙，官方给出的真实绘制坐标却是 **`(1012, 464)`**！
   - 比较 `(1012, 400)` 和 `(1012, 464)` 可以发现，它的 Y 增加了刚好一个物理步长 `step_y = 64`。代入我们的等距反推矩阵，这就意味着：在底层的渲染层面，它需要相对于它本该待的锚点向 X 轴正方向被强行偏移一个 `step_x`！
   - **铁证如山**：画师在切割 `/` 墙贴图时采用了截然不同的对齐方式，导致它在被传入游戏引擎时，必须附带 `offset_a_x = +90.0` 的补偿值才能严丝合缝。

## 6. 实验室长墙壁的“半步长网格吸附”与编辑器完美对接

In the following change, we integrate the Lab Wall snapping logic to ensure parity with the game's official editor.

### 1. 深入分析：官方编辑器的 45 × 32“半步长吸附网格”

官方编辑器对于标准墙体（40 × 28）采用的是 **40 × 28** 的全步长吸附网格。然而，对于实验室厚墙（90 × 64），为了提供更加细腻的半格错位或对齐能力，编辑器在底层使用的是**减半的吸附步长：45 × 32**。

当我们分析手动完美对齐的关卡数据（如 `lab_wall_long.json`）中四角立柱的绝对物理坐标时，可以得到如下分布：
* **P1**: `(112, 400)`
* **P2**: `(562, 720)`
* **P3**: `(562, 80)`
* **P4**: `(1012, 400)`

通过对这些坐标进行取模（Modulo）运算，我们发现了一个惊人的物理规律：
* $112 \pmod{45} = 22$ 且 $400 \pmod{32} = 16$
* $562 \pmod{45} = 22$ 且 $720 \pmod{32} = 16$
* $562 \pmod{45} = 22$ 且 $80 \pmod{32} = 16$
* $1012 \pmod{45} = 22$ 且 $400 \pmod{32} = 16$

这从数学上证明了，官方编辑器中实验室墙体的**固定物理吸附轨道**是：
* **`posX ≡ 22 (mod 45)`**
* **`posY ≡ 16 (mod 32)`**

此处的余数 `(22.5, 16)` 并不是随机的，它正是当前吸附步长的一半 `(45 / 2, 32 / 2)` 四舍五入后得到的吸附中心值。

### 2. 解决方案：强行轨道对齐（Orbit Alignment）

为了确保自动生成的实验室墙壁能够与后续的手动修改完美咬合，`Auto Mapper` 重构了网格原点（Shift）计算公式。当生成实验室墙体时，不再使用原始的 90 × 64 步长计算原点，而是强行将原点（Shift）吸附到 `22 (mod 45)` 和 `16 (mod 32)` 的轨道上。

#### 统一对齐算法实现：
* **X 轴原点计算**（基于 45 步长及 22.5 偏置）：
  $$\text{shift\_x} = \text{round}\left( \frac{\text{map\_size\_x} / 2.0 - 22.5}{45.0} \right) \times 45.0 + 22.5$$
* **Y 轴原点计算**（基于 32 步长及 16.0 偏置）：
  $$\text{shift\_y} = \text{round}\left( \frac{16.0 - 16.0}{32.0} \right) \times 32.0 + 16.0 + 64.0 = 80.0$$

通过这一轨道对齐重构，`Auto Mapper` 生成的实验室墙壁绝对物理坐标被强行规范在 `22 (mod 45)` 和 `16 (mod 32)` 上。当用户在官方编辑器中打开地图并按 `s` 开启网格吸附继续手动拉墙时，**手摆的新墙体会与自动生成的旧墙体处于同一条吸附轨道，实现 100% 完美的无缝拼接与混编修改**。
